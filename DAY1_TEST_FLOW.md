# Day 1 Test Flow: Complete Walkthrough

## Overview

This document traces the complete execution flow of `tests/test_simple.mlir` through the NN-MLIR compiler system, from source file to parsed and printed output.

## The Test File

**Location**: `tests/test_simple.mlir`

```mlir
// Simple test: matrix multiply followed by relu
func.func @simple_network(%input: tensor<2x3xf32>, %weight: tensor<3x4xf32>, %bias: tensor<4xf32>) -> tensor<2x4xf32> {
  // Dense operation: matrix multiply + bias
  %dense_result = nn.dense %input, %weight, %bias : tensor<2x3xf32>, tensor<3x4xf32>, tensor<4xf32> -> tensor<2x4xf32>

  // ReLU operation: activation
  %relu_result = nn.relu %dense_result : tensor<2x4xf32> -> tensor<2x4xf32>

  return %relu_result : tensor<2x4xf32>
}
```

### What This Represents

A neural network computation with two sequential operations:

1. **Dense Layer**: Takes three inputs
   - `%input`: 2×3 feature matrix
   - `%weight`: 3×4 weight matrix
   - `%bias`: 4-element bias vector
   - Computes: `output = input @ weight + bias` (matrix multiply + bias addition)
   - Returns: 2×4 result

2. **ReLU Activation**: Takes the dense output and applies ReLU
   - `relu(x) = max(0, x)` elementwise
   - Preserves tensor shape (2×4)
   - Returns: 2×4 result

## The Compilation Pipeline

### Step 1: Build Phase

Before any compilation can happen, the build system must generate the dialect code:

```bash
$ make
```

This executes the Makefile which:

1. **Runs TableGen** (from `NNOps.td`):
   ```bash
   mlir-tblgen --gen-dialect-decls --dialect=nn ... -o build/include/NN/NNDialect.h.inc
   mlir-tblgen --gen-dialect-defs --dialect=nn ... -o build/include/NN/NNDialect.cpp.inc
   mlir-tblgen --gen-op-decls ... -o build/include/NN/NNOps.h.inc
   mlir-tblgen --gen-op-defs ... -o build/include/NN/NNOps.cpp.inc
   ```

   This generates the C++ classes for:
   - `NNDialect` (the dialect container)
   - `DenseOp` class (represents `nn.dense`)
   - `ReluOp` class (represents `nn.relu`)

2. **Compiles dialect code**:
   ```bash
   clang++ ... lib/NN/NNDialect.cpp -o build/NNDialect.o
   clang++ ... lib/NN/NNOps.cpp -o build/NNOps.o
   ```

3. **Compiles and links nn-opt**:
   ```bash
   clang++ ... tools/nn-opt.cpp build/NNDialect.o build/NNOps.o ... -o nn-opt
   ```

### Step 2: Dialect Registration

When `nn-opt` starts, `main()` runs:

**File**: `tools/nn-opt.cpp`

```cpp
int main(int argc, char **argv) {
  mlir::DialectRegistry registry;

  // Register all standard MLIR dialects (func, tensor, etc.)
  mlir::registerAllDialects(registry);

  // Register our NN dialect
  registry.insert<mlir::nn::NNDialect>();

  return asMainReturnCode(
      mlir::MlirOptMain(argc, argv, "NN MLIR Optimizer", registry));
}
```

This tells MLIR:
- "When you see `nn.dense` or `nn.relu` operations in a file, use the NNDialect to parse/print them"
- The `NNDialect::initialize()` method (from NNDialect.cpp) registers both DenseOp and ReluOp

### Step 3: File Parsing

When you run:
```bash
$ ./nn-opt tests/test_simple.mlir
```

MLIR's parser does this:

1. **Read the file** into memory

2. **Parse the outer structure**:
   - Recognizes `func.func` is from the standard `func` dialect
   - Creates a function with name `@simple_network`
   - Records function signature: 3 inputs → 1 output

3. **Parse the function body**:
   - Sees `nn.dense` — looks up `nn` in the dialect registry
   - Finds `NNDialect`, asks it to parse the operation
   - Calls `DenseOp::parse()` (generated by TableGen) which:
     - Reads operands: `%input`, `%weight`, `%bias`
     - Reads types: `tensor<2x3xf32>, tensor<3x4xf32>, tensor<4xf32> -> tensor<2x4xf32>`
     - Creates a `DenseOp` instance in memory
     - Stores result value `%dense_result`

   - Sees `nn.relu` — repeats process:
     - Creates a `ReluOp` instance
     - Uses `%dense_result` as input
     - Stores result value `%relu_result`

   - Sees `return` — parses the return statement with `%relu_result`

4. **Build the in-memory IR**:
   ```
   Module
   └── FuncOp @simple_network
       ├── Block (function body)
       │   ├── DenseOp
       │   │   ├── operands: [%input, %weight, %bias]
       │   │   └── result: %dense_result (type: tensor<2x4xf32>)
       │   ├── ReluOp
       │   │   ├── operand: %dense_result
       │   │   └── result: %relu_result (type: tensor<2x4xf32>)
       │   └── ReturnOp
       │       └── operand: %relu_result
   ```

### Step 4: Verification (Semantic Checks)

After parsing, MLIR's type system verifies the IR is valid:

1. **DenseOp verification** (from NNOps.td):
   - Input types must be: `tensor<2x3xf32>, tensor<3x4xf32>, tensor<4xf32>` ✓
   - Output type must be: `tensor<2x4xf32>` ✓

2. **ReluOp verification**:
   - Input type must be: `tensor<2x4xf32>` ✓
   - Output type must be: `tensor<2x4xf32>` ✓

3. **Return verification**:
   - Return type matches function signature ✓

### Step 5: Printing (Round-Trip)

After parsing, `MlirOptMain` prints the IR back out:

Each operation's `print()` method (generated by TableGen) is called:

**DenseOp::print()**:
```
%0 = nn.dense %arg0, %arg1, %arg2 : tensor<2x3xf32>, tensor<3x4xf32>, tensor<4xf32> -> tensor<2x4xf32>
```

**ReluOp::print()**:
```
%1 = nn.relu %0 : tensor<2x4xf32> -> tensor<2x4xf32>
```

**Final output**:
```mlir
module {
  func.func @simple_network(%arg0: tensor<2x3xf32>, %arg1: tensor<3x4xf32>, %arg2: tensor<4xf32>) -> tensor<2x4xf32> {
    %0 = nn.dense %arg0, %arg1, %arg2 : tensor<2x3xf32>, tensor<3x4xf32>, tensor<4xf32> -> tensor<2x4xf32>
    %1 = nn.relu %0 : tensor<2x4xf32> -> tensor<2x4xf32>
    return %1 : tensor<2x4xf32>
  }
}
```

## What Each Component Did

| Component | File | Role |
|-----------|------|------|
| **TableGen** | `include/NN/NNOps.td` | Specification: Declares what DenseOp and ReluOp are |
| **Generated Code** | `build/include/NN/NNOps.{h,cpp}.inc` | Implementation: C++ classes for DenseOp and ReluOp |
| **Dialect** | `lib/NN/NNDialect.cpp` | Registration: Tells MLIR about the nn dialect and its operations |
| **nn-opt** | `tools/nn-opt.cpp` | Tool: Command-line interface that uses the dialect to parse/print MLIR |
| **Test File** | `tests/test_simple.mlir` | Input: Sample neural network in MLIR syntax |

## Why This Matters

This test demonstrates **proof that your compiler infrastructure works end-to-end**:

✅ **Parser works**: It correctly parsed `nn.dense` and `nn.relu` operations
✅ **Type system works**: It verified tensor types match
✅ **Dialect works**: It correctly registered and used the nn dialect
✅ **Printer works**: It converted the in-memory IR back to text form

**The round-trip property** (parse → print produces equivalent output) is critical because:
- If the parser is broken, the printed output would be wrong
- If the printer is broken, operations would be mangled
- If the dialect registration is broken, operations would fail to parse
